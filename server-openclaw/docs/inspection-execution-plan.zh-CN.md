# OpenClaw 巡检产品执行计划

## 1. 文档目标

这份文档把 3 件事合在一起：

- 一版可落地的巡检产品 PRD
- 一版基于当前后台系统的优化方案
- 一版命中引擎规则设计文档

目标不是再做一个 Demo，而是把当前 OpenClaw 巡检能力往“可持续接更多业务、可复制到更多门店、可给业务直接用”的垂域产品推进。

---

## 2. 当前项目现状

### 2.1 已有能力

当前项目已经有以下基础：

- 巡检执行链路已经跑通
- 企业微信对话入口已经接通
- 企业微信文档/表格沉淀链路已经具备基础能力
- 巡检后台已经上线，已有页面：
  - 巡检总览
  - 门店管理
  - 监控模块
  - 点检模板库
  - 巡检计划
  - 计划绑定
  - 执行记录
- 后端已经承担了门店、计划、绑定、任务、结果等基础数据管理
- OpenClaw 已经具备两种巡检模式：
  - 基准图巡检
  - 点检项描述巡检

### 2.2 当前主要问题

当前产品已经能跑，但离“业务稳定使用”还有几个关键差距：

- 用户输入命中还不够稳，门店名、计划名、点位名容易误判或没命中
- 结果输出虽然有了结构，但还没有完全做到“业务一眼能看懂”
- 批量巡检还没有形成标准产品体验
- 后台配置能力够用但不够顺手，别名、关键词、触发话术、批量任务等能力还不完整
- 系统还没有把“预计耗时”“任务进度”“失败重试”“命中日志”做成标准机制
- 新业务接入时，仍然比较依赖人工理解和现场调试

## 3. 巡检产品 PRD

## 3.1 产品定位

产品名称建议：

- OpenClaw 门店智能巡检助手

产品定位：

- 面向门店运营、督导、区域负责人和管理人员的智能巡检产品
- 让业务人员只需要说出门店和巡检计划，就能自动命中配置、执行巡检、输出业务结果并沉淀记录

一句话定义：

- 这是一个“会理解业务配置、会看监控画面、会输出巡检结果、会形成记录闭环”的巡检执行产品

## 3.2 产品目标

第一阶段产品目标：

- 让用户一句话稳定命中门店、巡检计划、监控点位
- 让巡检结果逐条输出每个点检项的结果、匹配度和原因
- 让后台、企业微信、巡检执行、文档沉淀形成标准闭环

第二阶段产品目标：

- 支持批量巡检
- 支持按区域、门店组、计划组批量执行
- 支持预计耗时、任务进度、异常升级、批量汇总展示

第三阶段产品目标：

- 把巡检产品做成可接入更多业务场景的垂域平台
- 支持更复杂的规则、更多业务场景模板、更多告警策略和更强的数据沉淀

## 3.3 核心用户

### 业务使用者

- 区域经理
- 巡店人员
- 门店运营
- 店长

他们的诉求：

- 快速触发巡检
- 结果看得懂
- 出异常能及时知道

### 配置维护者

- 平台运营
- 实施人员
- 项目管理员

他们的诉求：

- 能在后台快速配置门店、点位、计划、模板和触发词
- 能知道为什么没命中、为什么误命中

### 产品/技术维护者

- OpenClaw 运维人
- 前后端开发

他们的诉求：

- 能快速定位问题
- 能扩展更多业务
- 能减少一线调试成本

## 3.4 核心场景

### 场景 1：单门店即时巡检

用户在企业微信里发送：

- `@小智 执行巡检计划 成都逮虾户零食有鸣 实时检查计划`

系统完成：

- 命中门店
- 命中计划
- 命中绑定点位
- 抽帧并执行
- 输出逐条点检结果

### 场景 2：单门店基准图巡检

用户发送图片加流地址，或命中后台已绑定的基准图计划。

系统完成：

- 命中基准图巡检计划
- 抓取帧与基准图对比
- 输出差异结果和原因

### 场景 3：批量巡检

用户发送：

- `@小智 执行巡检计划 成都区域 营业画面点检`

系统完成：

- 识别区域内门店集合
- 批量生成任务
- 提示预计耗时
- 分批执行
- 汇总输出结果
- 写入企业微信表格

### 场景 4：定时巡检

系统在固定时间自动执行：

- 开店前巡检
- 营业中随机巡检
- 闭店后巡检

### 场景 5：异常复核

用户对异常门店发起：

- `@小智 复核 成都逮虾户零食有鸣 实时检查计划`

系统复用上次配置，快速重跑并补充结果。

## 3.5 产品原则

- 先命中，再执行
- 先规则，后模型
- 结果必须逐条展开
- 原因必须业务可懂
- 后台配置必须能被业务或运营持续维护
- 反馈必须给用户明确预期
- 企业微信机器人场景必须先 `@机器人`，再触发命中和执行

## 3.6 核心功能

### A. 配置中心

- 门店管理
- 监控点位管理
- 点检模板库
- 巡检计划管理
- 计划与点位绑定
- 告警对象配置
- 别名和关键词配置

### B. 执行中心

- 单店巡检
- 批量巡检
- 定时巡检
- 重试巡检
- 复核巡检
- 企业微信 `@机器人` 触发执行

### C. 命中中心

- 门店命中
- 计划命中
- 点位命中
- 别名命中
- 关键词命中
- 低置信度澄清
- 命中日志

### D. 结果中心

- 逐条点检项结果
- 匹配度
- 不通过原因
- 监控点位
- 巡检截图

### E. 记录中心

## 3.7 成功指标

建议第一阶段直接盯这几个指标：

- 门店命中准确率 >= 95%
- 巡检计划命中准确率 >= 95%
- 单次巡检平均首响时间 <= 1 秒
- 单点位巡检结果返回时间 <= 8 秒
- 批量巡检预计耗时误差 <= 30%
- 业务侧“结果看不懂”反馈数量持续下降
- 异常结果复核闭环率逐步提升

当前后台已经有以下页面：

- `DashboardView.vue` 对应巡检总览
- `StoresView.vue` 对应门店管理
- `StreamsView.vue` 对应监控模块管理
- `TemplateLibraryView.vue` 对应点检模板库
- `PlansView.vue` 对应巡检计划
- `BindingsView.vue` 对应计划绑定
- `JobsView.vue` 对应执行记录

后台优化不是单纯美化，而是为了支撑产品化：

- 让配置更完整
- 让命中更可控
- 让批量任务更容易发起和查看
- 让结果和问题更容易复盘

- 新手路径更短
- 表单字段更少但更准
- 有默认值、有模板、有推荐

## 5. 命中引擎规则设计文档

## 5.1 设计目标

命中引擎负责解决一句话到执行上下文的转换问题。

目标：

- 你说哪个门店，就尽量命中哪个门店
- 你说哪个计划，就执行哪个计划
- 不再被默认绑定关系“抢执行权”
- 命中不准时先澄清，不盲目执行

## 5.2 设计原则

- 规则优先，模型兜底
- 多实体独立打分，再联合决策
- 先精确命中，再模糊命中
- 绑定关系只在门店和计划确定后参与，不允许反向抢优先级
- 低置信度必须回问

## 5.3 命中对象

命中引擎至少处理 4 类对象：

- 门店
- 巡检计划
- 监控点位
- 批量范围

## 5.4 输入标准化

用户输入进来后，先做标准化：

- 去空格
- 去无意义符号
- 统一全角半角
- 识别并保留 `@小智`
- 统一触发前缀
- 识别固定指令前缀：`执行巡检计划`

这里要区分两个入口：

- 企业微信机器人入口：`@机器人` 是触发前置条件，不能省略
- OpenClaw 控制台入口：不需要 `@机器人`

标准化时的正确做法不是简单“去掉 @小智”，而是：

- 先识别消息是否已经 `@机器人`
- 如果是企业微信群场景且没有 `@机器人`，直接提示补充，不进入执行链路
- 在完成触发校验后，再从正文里剥离 `@机器人` 文本，进入后续门店/计划/点位命中

标准化结果里，拆成：

- 原始文本
- 清洗文本
- 触发动作
- 候选门店词
- 候选计划词
- 候选点位词
- 候选范围词

## 5.5 命中流程

### 第一步：动作识别

识别用户要干什么：

- 执行巡检
- 批量巡检
- 重试巡检
- 复核巡检
- 查看结果

企业微信场景下，动作识别前先做触发校验：

- 单聊机器人：按机器人接收规则直接进入识别
- 群聊机器人：必须先 `@机器人`
- 未 `@机器人`：不进入巡检命中，不执行任务

### 第二步：门店命中

匹配顺序：

1. 门店全称精确匹配
2. 门店别名精确匹配
3. 门店简称匹配
4. 门店模糊匹配
5. 模型在候选门店中做兜底判断

门店评分建议：

- 全称完全命中：100
- 别名完全命中：95
- 简称命中：88
- 模糊相似命中：70-85
- 模型兜底：单独给置信度，不直接覆盖强规则结果

### 第三步：计划命中

匹配顺序：

1. 计划全称精确匹配
2. 计划别名匹配
3. 计划关键词组匹配
4. 场景模板匹配
5. 模型候选判断

计划评分建议：

- 全称完全命中：100
- 别名完全命中：94
- 关键词完整命中：88
- 部分关键词命中：72-84

### 第四步：点位命中

如果用户明确说了点位，如“门头”“收银台”“卖场全景”，就优先命中点位。

顺序：

1. 点位全称
2. 点位别名
3. 场景类型映射
4. 绑定关系候选过滤

### 第五步：绑定确认

只有当“门店 + 计划”都命中后，才去查绑定。

绑定层职责：

- 找出这个门店下能执行该计划的点位
- 如果用户指定点位，就优先执行指定点位
- 如果未指定点位，就按绑定优先级选择
- 如果一个计划绑定多个点位，则生成多点位任务

### 第六步：决策

联合评分后输出：

- 直接执行
- 提示澄清
- 拒绝执行并给候选项

## 5.6 命中优先级规则

强制优先级建议如下：

1. 用户明确输入的门店名
2. 用户明确输入的计划名
3. 用户明确输入的点位名
4. 后台绑定关系
5. 默认计划
6. 模型兜底

关键规则：

- 绑定关系不能在“计划未明确命中”时反推计划
- 默认计划只能在用户没有明确说计划时使用
- 模型不能越权替换一个已经高置信度命中的计划

## 5.7 置信度与澄清机制

建议设三档：

- `>= 0.90`：直接执行
- `0.75 - 0.89`：先简短确认，再执行
- `< 0.75`：不执行，给候选项

澄清示例：

- `我理解你要执行的是“成都逮虾户零食有鸣 - 实时检查计划”，如果不是，请回复候选编号。`

候选示例：

- `未完全命中，请确认：1. 成都逮虾户零食有鸣 2. 成都小智零食有鸣`

## 5.8 多关键词命中设计

支持以下输入模式：

- `@小智 执行巡检计划 成都逮虾户零食有鸣 实时检查计划`
- `@小智 执行巡检计划 成都逮虾户零食有鸣 收银台巡检`
- `@小智 执行巡检计划 成都区域 营业画面点检`
- `@小智 复核 成都逮虾户零食有鸣 实时检查计划`

建议支持的关键词类型：

- 动作词：执行、启动、运行、复核、重试、查看
- 对象词：门店、收银台、门头、卖场、仓库
- 计划词：实时检查、营业画面、基准图巡检、点检项巡检
- 范围词：区域、全部门店、今天、异常门店

## 5.9 实时同步机制

为了保证“后台改了门店和计划，对话侧能尽快生效”，建议做两层同步：

### 配置实时拉取

- 每次执行命中前先拉取最新配置摘要
- 或者使用短期缓存，缓存时间 30-120 秒

### 版本号机制

后台输出：

- 配置版本号
- 更新时间

OpenClaw 本地缓存记录：

- 当前版本号
- 上次刷新时间

如果版本号变化，立即刷新：

- 门店列表
- 计划列表
- 点位列表
- 绑定关系

## 5.10 命中日志设计

- 原始输入
- 标准化输入
- 候选门店列表
- 最终命中门店
- 候选计划列表
- 最终命中计划
- 候选点位列表
- 最终命中点位

## 6. 批量巡检产品方案

## 6.1 触发方式

建议支持以下几类批量入口：

- 区域批量巡检
- 门店组批量巡检
- 计划组批量巡检
- 异常门店复核

## 6.2 批量任务执行方式

批量任务不建议一次性同步跑完后再回结果，建议分成 3 段：

### 第一段：秒回受理

示例：

- `已接收，本次将巡检 18 家门店、42 个监控点、126 条点检项，预计 2-4 分钟完成。`

### 第二段：过程反馈

示例：

- `当前进度：已完成 7/18 家门店，发现 2 家异常。`

### 第三段：汇总反馈

示例：

- `批量巡检已完成：18 家门店中 13 家通过，3 家未通过，2 家待复核。已同步到企业微信表格。`

## 6.3 批量结果展示

建议结合现有后台和企业微信沉淀方式：

### 企业微信聊天里展示

- 汇总结论
- 异常门店列表
- 表格链接
- 关键截图

### 后台执行记录里展示

- 一个批次任务
- 批次下展开门店子任务
- 每个门店再展开点位结果

### 企业微信表格里展示

一行一条门店巡检结果，字段建议包括：

- 巡检时间
- 门店名称
- 巡检计划
- 监控点位
- 巡检结论
- 通过项数
- 未通过项数
- 待复核项数
- 缩略图
- 巡检图片链接
- 异常摘要

## 7. 回复速度与用户预期设计

## 7.1 首响策略

继续保留“先秒回已接收，再补结果”的设计。

推荐文案：

- `已接收，正在按门店场景抓取流画面巡检。`

## 7.2 预计耗时策略

建议引入 ETA 计算：

- 基础耗时
- 门店数量
- 点位数量
- 点检项数量
- 并发数
- 当前队列长度

可先用简单估算公式：

```text
预计耗时 = 基础启动耗时
        + (门店数 × 点位数 × 单点位抽帧耗时) / 并发数
        + (点检项总数 × 单项分析耗时) / 并发数
        + 当前排队补偿
```

用户看到的文案建议做分档：

- `< 10 秒`：预计 2-8 秒
- `10-60 秒`：预计 10-60 秒
- `> 60 秒`：预计 1-4 分钟

## 7.3 加速策略

- 常用流地址首帧缓存
- 配置摘要缓存
- 模板解析缓存
- 批量任务并发执行
- 基准图计划与描述式计划分通道执行

## 8. 企业微信机器人触发规范

## 8.1 入口定义

当前巡检产品的主入口是企业微信机器人。

因此在产品定义上要明确：

- 企业微信群聊入口
- 企业微信单聊入口
- OpenClaw 控制台入口

其中业务侧主要使用的是：

- 企业微信群聊 `@小智`
- 企业微信单聊小智

## 8.2 触发规则

### 企业微信群聊

必须满足以下条件才触发：

- 消息里明确 `@小智`
- `@小智` 后面包含有效正文
- 正文命中巡检动作

未满足时的处理：

- 没有 `@小智`：不触发，不执行
- 只 `@小智` 没正文：返回引导话术
- 正文不完整：返回补充提示

### 企业微信单聊

单聊场景可以不依赖 `@小智`，直接进入动作识别。

### OpenClaw 控制台

控制台入口不依赖 `@小智`，但建议仍然保持和企微一致的指令格式，方便统一心智。

## 8.3 推荐输入模板

### 单店巡检

- `@小智 执行巡检计划 成都逮虾户零食有鸣 实时检查计划`

### 指定点位巡检

- `@小智 执行巡检计划 成都逮虾户零食有鸣 收银台巡检`

### 批量巡检

- `@小智 执行巡检计划 成都区域 营业画面点检`

### 异常复核

- `@小智 复核 成都逮虾户零食有鸣 实时检查计划`

## 8.4 预回复规范

企业微信机器人必须先给用户明确反馈，避免“像没收到一样”。

### 单店任务预回复

推荐文案：

- `已接收，正在按门店场景抓取流画面巡检，通常需要 2-6 秒。`

### 批量任务预回复

推荐文案：

- `已接收，正在生成批量巡检任务。`
- `本次预计巡检 18 家门店、42 个监控点、126 条点检项，预计 2-4 分钟完成。`

### 文档/表格写入类预回复

推荐文案：

- `已接收，正在整理并写入企业微信表格，通常需要 3-8 秒。`

## 8.5 过程反馈规范

### 单店任务

单店任务默认不做频繁过程播报，只保留：

- 已接收
- 最终结果

### 批量任务

批量任务建议增加阶段性进度反馈：

- `当前进度：已完成 7/18 家门店，发现 2 家异常。`

## 8.6 最终结果反馈规范

企业微信侧最终结果必须包含：

- 命中的门店
- 命中的巡检计划
- 命中的监控点位
- 巡检结论
- 点检项统计
- 逐条点检结果
- 不通过原因
- 巡检截图
- 表格/文档链接（如有）

## 8.7 异常输入处理规范

### 未 @机器人

- 群聊中不触发
- 不执行巡检

### 只 @机器人无正文

返回：

- `已收到，请在 @我 后补充门店名称和巡检计划，例如：@小智 执行巡检计划 成都逮虾户零食有鸣 实时检查计划`

### 命中失败

返回候选项，不直接执行。

### 多义命中

先澄清，再执行。

## 8.8 验收标准

企业微信机器人触发规范完成后，应满足：

- 群聊中只有 `@小智` 才会触发巡检
- 没有 `@小智` 不误触发
- 单店任务有预回复
- 批量任务有预计耗时和进度反馈
- 结果反馈格式统一

# OpenClaw 巡检产品推进总图

## 1. 课题定义

- 课题名称
  - OpenClaw 门店智能巡检产品化落地
- 核心目标
  - 让业务只说门店名称和巡检计划，系统自动命中后台配置并完成巡检执行
  - 让输出结果从“AI回答”升级成“业务可执行结果”
  - 让巡检形成从输入、执行、告警、留痕到复盘的完整闭环
- 业务价值
  - 降低人工重复截图、比对、登记成本
  - 统一巡检标准
  - 提升异常反馈速度
  - 沉淀结构化巡检数据资产

## 2. 课题推进如何又快又稳

### 2.1 方法论总原则

- 先闭环，再做厚
  - 先跑通一条最小业务链路
  - 再逐步补强识别率、速度、体验、报表
- 先命中，再执行
  - 先命中门店
  - 再命中巡检计划
  - 再命中监控模块
  - 最后才执行巡检
- 先标准化，再智能化
  - 先把门店、流地址、计划、点检项、绑定关系结构化
  - 再接 AI 执行与识别
- 先业务话术，再技术话术
  - 输出先让业务看懂
  - 再考虑技术字段沉淀
- 先留痕，再优化
  - 每次执行必须有记录
  - 每次异常必须可回查
  - 每次识别偏差必须可复盘

### 2.2 高效落地四步法

- 第一步：做最小可演示闭环
  - 后台可配置门店
  - 后台可配置监控模块
  - 后台可配置巡检计划
  - 后台可绑定计划到模块
  - 企业微信一句话可触发
  - 系统可返回结果
- 第二步：把链路标准化
  - 固化输入模板
  - 固化命中逻辑
  - 固化结果模板
  - 固化告警模板
- 第三步：把体验做流畅
  - 先秒回已接收
  - 再补最终结果
  - 首帧缓存
  - 常用门店和计划快捷命中
- 第四步：把结果沉淀成管理抓手
  - 执行记录
  - 企业微信文档
  - 智能表格
  - 日报与周报

### 2.3 为什么这样推进最快

- 快在不同时做所有事
  - 先只做一条业务场景
  - 先只做一个门店、一条流、一个计划
  - 先演示成功，再扩门店和计划
- 快在减少认知切换
  - 后台负责配置
  - OpenClaw 负责执行
  - 企业微信负责触达
  - 文档负责沉淀
- 快在复用已有能力
  - 复用 OpenClaw 消息链路
  - 复用流媒体抽帧
  - 复用企业微信插件
  - 复用智能表格沉淀

### 2.4 为什么这样推进更稳

- 稳在每一层都可验证
  - 后台能查配置
  - API 能查上下文
  - 执行日志能回放
  - 结果文档能验收
- 稳在先做结构化映射
  - 不是把所有理解交给 AI 自由发挥
  - 而是先通过后台映射把“门店、计划、模块”钉死
- 稳在允许人工复核
  - 对不稳定识别项允许进入待复核
  - 不硬判通过

## 3. 产品全流程

### 3.1 后台配置流程

- 门店管理
  - 创建门店
  - 录入门店名称、编码、区域、负责人
- 监控模块管理
  - 为门店创建多个监控模块
  - 每个模块绑定流地址
  - 每个模块配置模块名和别名
  - 可配置基准图
- 巡检计划管理
  - 创建巡检计划
  - 录入计划名称
  - 录入点检项
  - 录入匹配阈值
  - 录入计划类型
- 计划绑定
  - 绑定门店
  - 绑定监控模块
  - 绑定巡检计划

### 3.2 用户触发流程

- 用户输入
  - `@小智 成都小智零食有鸣 执行实时检查计划`
  - `@小智 成都小智零食有鸣 执行门店基准图巡检`
- 系统解析
  - 提取门店名称关键词
  - 提取巡检计划关键词
- 系统命中
  - 门店命中
  - 计划命中
  - 模块命中
  - 流地址命中
- 系统执行
  - 先秒回“已接收，正在巡检”
  - 抓取第一帧或随机帧
  - 执行基准图巡检或文字巡检
- 系统回传
  - 返回巡检结论
  - 返回命中的门店、计划、监控模块
  - 返回每条点检项结果
  - 返回巡检图片
- 系统沉淀
  - 写入执行记录
  - 写入企业微信文档
  - 写入企业微信智能表格

### 3.3 OpenClaw 执行过程

- 输入层
  - 企业微信消息
  - OpenClaw 对话
- 配置查询层
  - `GET /api/stores`
  - `GET /api/inspection-plans`
  - `GET /api/store-plan-bindings`
  - `GET /api/store-streams`
- 执行上下文层
  - `POST /api/execute/context`
  - 生成本次执行上下文
- 任务落库层
  - `POST /api/manual-executions`
- 巡检执行层
  - 流媒体抽帧
  - 基准图比对
  - 文字点检分析
- 结果回写层
  - `POST /api/inspection-results`
- 查询展示层
  - `GET /api/inspection-jobs`
  - `GET /api/inspection-results/:id`

## 4. 结果输出应该长什么样

### 4.1 业务侧结果模板

- 结果头部
  - 巡检结论
  - 门店
  - 巡检计划
  - 监控模块
  - 抽帧方式
  - 巡检时间
- 逐条点检结果
  - 点检项名称
  - 匹配结果
  - 匹配度
  - 不符合原因
  - 画面证据
- 结果尾部
  - 建议动作
  - 巡检图片
  - 文档记录链接

### 4.2 标准输出格式

- 点检项名称
  - 门口装修范围内保持干净整洁
- 匹配结果
  - 未通过
- 匹配度
  - 31%
- 不符合原因
  - 门口区域发现纸箱和杂物堆放
- 画面证据
  - 左下角识别到堆放纸箱，门口通道非净空状态

### 4.3 业务看得懂的文案标准

- 不写
  - 模型不稳定
  - 无法确定
  - 视觉结果异常
- 要写
  - 门口未发现营业时间牌
  - 收银台区域未识别到值班人员
  - 卖场区域发现杂物堆放
  - 电视设备未识别到开启画面

## 5. 核心功能开发坑位图

### 5.1 坑位一：用户说了门店名，但系统没命中

- 根因
  - 后台门店名和业务口语不一致
  - 没有配置别名
  - 系统只做了模糊匹配，没有做业务优先级
- 复盘
  - 仅靠 AI 理解门店名不稳
  - 必须有后台门店映射
- 解决方案
  - 门店名称标准化
  - 支持门店别名
  - 优先精确匹配，再模糊匹配

### 5.2 坑位二：命中了门店，但跑错监控模块

- 根因
  - 一个门店多路流
  - 没有绑定关系
  - 默认走了错误模块
- 复盘
  - 门店和流地址不能是一对一思维
  - 必须是一店多模块
- 解决方案
  - 增加监控模块概念
  - 计划绑定具体模块
  - 结果里回传模块名称

### 5.3 坑位三：结果只给总分，不给逐条点检项

- 根因
  - 技术输出导向
  - 没有以业务验收为目标设计返回模板
- 复盘
  - 业务不关心模型摘要
  - 业务关心哪一条没过
- 解决方案
  - 逐条输出点检项
  - 逐条输出匹配度
  - 逐条输出不符合原因

### 5.4 坑位四：原因写得太技术，业务看不懂

- 根因
  - 直接暴露模型原始文案
  - 缺少业务语义转换层
- 复盘
  - 技术可读不等于业务可读
- 解决方案
  - 增加业务文案模板
  - 把技术判断改写成现场描述

### 5.5 坑位五：点检项巡检还带差异图

- 根因
  - 描述巡检和基准图巡检共用了一套产物结构
- 复盘
  - 两种巡检类型不能用同一种结果展示模板
- 解决方案
  - 计划一显示基准图和差异图
  - 计划二只显示巡检帧

### 5.6 坑位六：返回条目被截断

- 根因
  - 结果链路中做了条数上限截断
  - 文案过长导致前端或聊天端不友好
- 复盘
  - “12条封顶”会直接损伤业务信任
- 解决方案
  - 后端不截断
  - 聊天端按完整清单输出
  - 长内容在文档端完整沉淀

### 5.7 坑位七：企业微信多人可 @ 和文档写入互相打架

- 根因
  - 把聊天入口和文档授权压在同一条授权链路
- 复盘
  - 聊天和文档最好拆成双链路
- 解决方案
  - 长连接机器人负责回复
  - 企业应用直连负责写文档

### 5.8 坑位八：体验慢，业务以为没反应

- 根因
  - 直接等全流程完成才回复
  - 抽帧和识别耗时波动大
- 复盘
  - 用户感知比真实耗时更重要
- 解决方案
  - 先秒回已接收
  - 再补结果
  - 常用流首帧缓存

## 6. 实战复盘

### 6.1 这次做对了什么

- 把门店、模块、计划、绑定关系做成了后台配置
- 把企业微信触发做成了自然语言入口
- 把结果沉淀做成了文档和表格
- 把一店多模块跑通了
- 把计划一和计划二分开了

### 6.2 这次最值钱的经验

- 经验一
  - AI 能力一定要挂在结构化配置之上
- 经验二
  - 最小闭环比大而全更重要
- 经验三
  - 业务验收口径必须前置
- 经验四
  - 结果模板比模型参数更影响业务满意度

### 6.3 这次踩过的典型问题

- 多路流导致计划跑偏
- 文档权限和机器人回复互斥
- 点检项输出过短
- 原因解释过于技术化
- 描述巡检误带差异图
- 前端管理后台布局和体验仍需继续打磨

## 9. 给别人讲这套产品时怎么讲

### 9.1 一句话版本

- OpenClaw 不是在回答问题，而是在执行一套门店巡检流程

### 9.2 三句话版本

- 业务只需要说门店名称和巡检计划
- 系统自动去后台命中门店、计划和监控模块并完成巡检
- 最终输出逐条点检结果、异常原因和巡检记录沉淀

  - 后台确认 `实时检查计划` 已绑定到 `03收银台1`

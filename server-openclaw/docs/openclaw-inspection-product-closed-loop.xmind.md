# OpenClaw 巡检产品闭环

## 1. 产品目标

- 让业务人员只需要说一句话，就能让 OpenClaw 自动命中门店、监控模块、巡检计划并输出业务可读结果
- 让巡检结果不再停留在“回答一句话”，而是形成可执行、可追踪、可复盘的业务闭环
- 让后台配置、聊天触发、巡检执行、结果回传、文档沉淀统一成一套标准产品流程

## 2. 业务侧最终体验

- 业务输入
  - 在企业微信里直接说门店名称和巡检计划
  - 例如：`@小智 成都小智零食有鸣 执行实时检查计划`
  - 例如：`@小智 成都小智零食有鸣 执行门店基准图巡检`
- 系统自动完成
  - 识别门店名称
  - 识别巡检计划名称
  - 命中后台配置的监控模块
  - 抽取对应流媒体帧
  - 按巡检计划逐条执行点检
  - 输出业务可读结论
  - 写入巡检记录和企业微信文档/表格
- 业务看到的结果
  - 巡检结论
  - 命中的门店
  - 命中的监控模块
  - 命中的巡检计划
  - 每一条点检项的结果
  - 每一条点检项的匹配度
  - 如果不符合，明确说明原因
  - 巡检图片
  - 异常时的提醒动作

## 3. 产品核心原则

- 原则一：先命中，再执行
  - 不允许用户输入没有先匹配门店和计划就直接执行
  - 先做结构化命中，再做视觉巡检
- 原则二：结果必须逐条展开
  - 不能只给一句“通过/不通过”
  - 必须逐条列出点检项
- 原则三：原因必须业务可懂
  - 不写“模型判断失败”
  - 改成“门口未发现营业时间牌”
  - 改成“收银台区域未识别到值班人员”
  - 改成“卖场区域发现杂物堆放”
- 原则四：后台配置必须支持业务自己维护
  - 门店名称
  - 门店别名
  - 监控模块
  - 巡检计划
  - 点检项
  - 绑定关系
- 原则五：业务操作要傻瓜化
  - 业务不需要知道流地址
  - 业务不需要知道接口名
  - 业务只需要说门店名和计划名

## 4. 产品全链路

### 4.1 第一步：后台建档

- 门店管理
  - 在后台创建门店
  - 录入门店名称
  - 录入门店编码
  - 录入区域
  - 录入负责人
- 监控模块管理
  - 为一个门店创建多个监控模块
  - 每个监控模块绑定一个流地址
  - 每个监控模块有自己的模块名称
  - 例如：`01门头1`
  - 例如：`03收银台1`
  - 例如：`04卖场全景1`
- 巡检计划管理
  - 创建巡检计划
  - 配置巡检类型
  - 配置点检项列表
  - 配置阈值
- 计划绑定
  - 把巡检计划绑定到具体门店的具体监控模块
  - 同一个门店可以有多个计划
  - 同一个门店不同模块可以绑定不同计划

### 4.2 第二步：用户发起巡检

- 用户入口
  - 企业微信群聊
  - 企业微信单聊
  - OpenClaw 对话框
- 推荐输入模板
  - `@小智 成都小智零食有鸣 执行实时检查计划`
  - `@小智 成都小智零食有鸣 执行门店基准图巡检`
  - `@小智 成都小智零食有鸣 执行收银台规范巡检`
- 产品要求
  - 用户输入里至少要有门店名称
  - 用户输入里至少要有巡检计划名称
  - 如果缺任一项，先返回补充提示，不直接执行

### 4.3 第三步：系统命中后台配置

- 输入解析层
  - 提取门店关键词
  - 提取巡检计划关键词
  - 识别用户是否在执行计划一还是计划二
- 配置命中层
  - 调用 `GET /api/stores`
  - 调用 `GET /api/inspection-plans`
  - 调用 `GET /api/store-plan-bindings`
  - 调用 `GET /api/store-streams`
- 命中规则
  - 先按门店名称精确匹配
  - 再按门店别名匹配
  - 再按计划名称精确匹配
  - 再按计划别名或关键词匹配
  - 找到门店和计划后，再去绑定关系里确认具体监控模块
- 命中结果
  - 命中门店
  - 命中巡检计划
  - 命中监控模块
  - 命中流地址
- 命中失败时的产品表现
  - 门店未命中：提示可选门店名
  - 计划未命中：提示该门店可执行计划
  - 门店和计划都有，但无绑定：提示“该门店暂未绑定此计划”

### 4.4 第四步：OpenClaw 执行巡检

- 执行上下文生成
  - 调用 `POST /api/execute/context`
  - 组装门店、计划、监控模块、流地址、点检项
- 巡检执行
  - 文字点检：从流媒体抓帧后逐条点检
  - 基准图巡检：从流媒体抓帧后与基准图做比对
- 执行结果落库
  - 调用 `POST /api/manual-executions`
  - 调用 `POST /api/inspection-results`
- 巡检记录查询
  - 调用 `GET /api/inspection-jobs`
  - 调用 `GET /api/inspection-results/:id`

### 4.5 第五步：结果回传给业务

- 第一段反馈
  - 秒回：已接收，正在巡检
- 第二段反馈
  - 回传最终结果
  - 回传巡检图片
  - 回传逐条点检项结果
  - 回传异常原因
- 第三段沉淀
  - 写入企业微信文档
  - 写入企业微信智能表格
  - 形成当天巡检记录

## 5. 输出结果标准

### 5.1 业务可读输出模板

- 巡检结论
  - 通过
  - 未通过
  - 待人工复核
- 命中信息
  - 门店
  - 监控模块
  - 巡检计划
  - 抽帧方式
  - 巡检时间
- 逐条点检结果
  - 点检项名称
  - 匹配结果
  - 匹配度
  - 不符合原因
  - 观察证据
- 巡检图片
  - 当前巡检帧
- 后续动作
  - 是否需要人工复核
  - 是否需要通知负责人
  - 是否需要写入日报

### 5.2 逐条点检项输出格式

- 每条点检项统一格式
  - `点检项：门口营业时间牌在位`
  - `结果：未通过`
  - `匹配度：22%`
  - `原因：门口区域未发现营业时间牌`
  - `证据：画面中仅识别到玻璃门和促销海报，未识别到营业时间牌`
- 每条点检项都必须有结果
  - 通过
  - 未通过
  - 待复核
- 每条点检项都必须有原因
  - 不能只写“未通过”
  - 必须解释不通过原因

### 5.3 业务结果示例

- 示例标题
  - `成都小智零食有鸣 - 实时检查计划 - 03收银台1`
- 示例结果
  - `巡检结论：未通过`
  - `门店：成都小智零食有鸣`
  - `监控模块：03收银台1`
  - `巡检计划：实时检查计划`
  - `巡检项统计：共 12 项，通过 7 项 / 未通过 3 项 / 待复核 2 项`
- 示例逐条输出
  - `1. 门口装修范围内保持干净整洁`
  - `结果：未通过`
  - `匹配度：31%`
  - `原因：门口区域发现纸箱和杂物堆放`
  - `2. 门店外未堆放废纸杂物`
  - `结果：未通过`
  - `匹配度：18%`
  - `原因：门口右侧发现拖把和纸皮堆放`
  - `3. 收银台有值班人员`
  - `结果：通过`
  - `匹配度：91%`
  - `原因：收银台区域识别到一名站立人员`
  - `4. 收银区域干净整洁`
  - `结果：待复核`
  - `匹配度：58%`
  - `原因：画面部分区域被遮挡，无法确认台面是否完全整洁`

## 6. 后台怎么配置这套产品

### 6.1 门店配置

- 后台页面
  - 门店管理
- 要配置的字段
  - 门店名称
  - 门店编码
  - 区域
  - 负责人
- 配置目标
  - 保证业务输入的门店名可以被系统命中

### 6.2 监控模块配置

- 后台页面
  - 监控模块
- 要配置的字段
  - 所属门店
  - 监控模块名称
  - 流地址
  - 模块别名
  - 基准图
- 配置目标
  - 保证系统知道这个计划要巡哪一路监控

### 6.3 巡检计划配置

- 后台页面
  - 巡检计划
- 要配置的字段
  - 计划名称
  - 计划类型
  - 点检项
  - 阈值
- 配置目标
  - 保证系统命中计划后知道要检查什么

### 6.4 计划绑定配置

- 后台页面
  - 计划绑定
- 要配置的字段
  - 门店
  - 监控模块
  - 巡检计划
- 配置目标
  - 保证用户说“某门店执行某计划”时，系统能落到具体监控位

## 7. 业务操作示范

### 7.1 业务问：小智线上巡检应该怎么做

- 操作示范
  - 第一步：打开后台
  - 第二步：进入门店管理，确认门店 `成都小智零食有鸣` 已创建
  - 第三步：进入监控模块，确认该门店下已有 `01门头1`、`03收银台1`、`04卖场全景1`
  - 第四步：进入巡检计划，确认已有 `实时检查计划`
  - 第五步：进入计划绑定，确认 `实时检查计划` 已绑定到 `03收银台1`
  - 第六步：去企业微信对小智说：`@小智 成都小智零食有鸣 执行实时检查计划`
  - 第七步：小智先回复“已接收，正在巡检”
  - 第八步：小智返回门店、计划、监控模块、巡检图片和逐条点检结果
  - 第九步：后台执行记录同步新增一条任务
  - 第十步：企业微信文档/表格同步新增一条巡检记录

### 7.2 业务操作话术模板

- 标准模板
  - `@小智 门店名称 执行巡检计划名称`
- 示例一
  - `@小智 成都小智零食有鸣 执行实时检查计划`
- 示例二
  - `@小智 成都小智零食有鸣 执行门店基准图巡检`
- 示例三
  - `@小智 成都小智零食有鸣 执行收银台规范巡检`

### 7.3 业务验收标准

- 验收一：门店必须命中正确
- 验收二：巡检计划必须命中正确
- 验收三：监控模块必须命中正确
- 验收四：输出必须逐条列出点检项
- 验收五：每条点检项必须给匹配度
- 验收六：每条点检项必须给原因
- 验收七：后台执行记录必须可查
- 验收八：文档沉淀必须可查

## 8. 产品闭环定义

- 闭环一：配置闭环
  - 后台能建门店
  - 后台能建监控模块
  - 后台能建巡检计划
  - 后台能建绑定关系
- 闭环二：输入闭环
  - 用户一句话能命中门店
  - 用户一句话能命中巡检计划
  - 系统能找到对应监控模块
- 闭环三：执行闭环
  - 能抽帧
  - 能分析
  - 能逐条输出点检结果
- 闭环四：结果闭环
  - 企业微信有结果
  - 后台有执行记录
  - 文档有巡检记录
- 闭环五：管理闭环
  - 业务知道是哪家门店
  - 业务知道是哪一路监控
  - 业务知道是哪一条点检项没过
  - 业务知道为什么没过
  - 业务知道下一步该找谁处理

## 9. 当前产品最重要的改造要求

- 要求一：命中逻辑前置
  - 所有巡检都必须先命中门店和计划
- 要求二：输出从“摘要”升级为“逐条结果”
  - 不能只给总分
  - 必须给每一条点检项结果
- 要求三：原因从“技术话术”升级为“业务话术”
  - 不写“视觉模型不稳定”
  - 改写成“门口区域未发现营业时间牌”
- 要求四：结果模板标准化
  - 所有巡检结果统一格式
  - 方便业务看
  - 方便文档沉淀
- 要求五：后台和聊天入口必须一致
  - 后台配什么
  - 聊天里就能执行什么

## 11. 一句话总结

- OpenClaw 不是回答一个问题，而是执行一套完整巡检流程
- 业务只需要说门店和计划，系统自动去后台命中配置并完成巡检
- 最终输出必须是业务看得懂、能处理、能追踪、能沉淀的产品级结果
